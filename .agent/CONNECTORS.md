# CONNECTORS.md — как Codex должен использовать коннекторы

Цель коннекторов — дать Codex факты и действия, а не просто “подумать”.

## Общий алгоритм

1. Определи, какие внешние системы реально нужны.
2. Получи только нужный минимум данных.
3. Перед destructive действиями запроси подтверждение.
4. После действия проверь результат.
5. В отчёте укажи, какой коннектор использовался и зачем.

---

## GitHub

Используй для:

- чтения репозитория;
- поиска похожих проектов;
- анализа issues/PR;
- создания ветки;
- коммита;
- PR;
- code review;
- CI logs.

Перед изменениями:

- проверь текущую ветку;
- проверь незакоммиченные изменения;
- не перетирай чужую работу.

---

## Vercel

Используй для:

- проверки деплоя;
- логов build/runtime;
- env vars;
- preview URL;
- production diagnostics.

Нельзя без подтверждения:

- менять production env;
- удалять проект;
- менять домены;
- делать rollback production;
- менять billing/teams.

---

## Supabase

Используй для:

- schema inspection;
- migrations;
- RLS;
- auth;
- storage;
- edge functions;
- logs.

Нельзя без подтверждения:

- DROP;
- массовый DELETE;
- отключение RLS;
- изменение auth policies;
- irreversible migrations.

Всегда делай SQL migration, а не ручное изменение “в голове”.

---

## Figma / Canva

Используй для:

- извлечения UI references;
- генерации макетов;
- согласования визуального стиля;
- подготовки креативов;
- лендингов и презентаций.

---

## Gmail / Calendar / Contacts

Используй только если задача явно про коммуникации, встречи, follow-up, отправку писем или поиск контактов.

Не отправляй письма без явного указания пользователя “отправь”.

---

## OpenAI / Claude / AI agents

Используй AI-агентов для:

- генерации черновиков;
- классификации;
- сравнения вариантов;
- code review;
- контентных задач;
- извлечения данных;
- проверки качества.

Не используй AI там, где лучше обычный код, SQL, regex, cron или deterministic parser.
